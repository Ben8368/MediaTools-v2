# Legacy UI Compatibility Baseline

本文档记录 MediaTools v2 轻前端的兼容原则。目标不是复制 Legacy 的复杂实现，而是保留用户熟悉的布局、视觉节奏和操作路径，让新版本在重构后仍有最小学习成本。

## 1. 当前结论

- v2 仓库中的 `frontend/` 目前只有 `.vite`、`dist`、`node_modules` 等未跟踪残留，没有可作为基线的源码或 `package.json`。
- `.agents/` 目前为空是正常状态；项目统一入口仍是根目录 `AGENTS.md`。除非后续需要放置不同 AI 工具的专属补充规则，否则无需强行填充。
- 轻前端技术栈必须先参考 Legacy。若 Legacy 使用 Vite / React / TypeScript 或相近栈，v2 优先沿用相近方案，便于恢复 UI 样式和组件习惯。
- Legacy 前端考古确认旧版使用 Vite + React + TypeScript + Vitest，包含桌面式左侧导航、窗口工作区、右侧状态面板和下载工作台组件。v2 采用同类技术栈，但只迁移壳层与下载工作台体验，不迁移旧 API 耦合、AI/PS/AE、文件管理和 vendor/构建产物。

## 2. 前端兼容原则

- **布局优先兼容**：技术栈可以变化，但主导航、工具分区、任务列表、输出结果区和设置入口应尽量保持 Legacy 的空间关系。
- **业务逻辑不进前端**：前端只负责参数填写、任务提交、状态展示和结果查看；媒体处理仍通过 Python core / CLI / API 适配层完成。
- **下载工作台优先**：Phase 3 的首个前端界面优先服务下载功能，而不是泛化首页或营销式页面。
- **渐进恢复**：先恢复最高频工作流的界面骨架，再补视觉细节、快捷操作和批处理能力。
- **可替换技术栈**：将颜色、间距、字体、密度、面板尺寸等沉淀为设计 token 或 CSS 变量，避免未来换前端框架时重写用户体验。

## 3. Legacy 考古清单

实施轻前端前，应先从 Legacy 仓库确认：

- 前端技术栈：框架、构建工具、组件库、样式方案、桌面壳方案。
- 页面结构：导航位置、工具入口、主工作区、任务队列、结果预览、设置入口。
- 下载流程：URL 输入、批量导入、字幕选项、输出目录、进度展示、失败重试。
- 视觉规则：主色、背景、字体层级、间距、表格/列表密度、按钮和表单风格。
- 可复用资产：图标、logo、空状态、常用文案。只迁移自有资产，不迁移 vendor 或构建产物。

## 4. 轻前端 MVP 范围

首个轻前端版本建议只覆盖：

- 下载 URL 输入与批量列表导入。
- 人工字幕 / 自动字幕、字幕语言、输出目录等下载参数。
- dry-run 预览与实际下载触发。
- 任务队列、状态、错误摘要、输出文件入口。
- doctor 环境状态展示：ffmpeg、ffprobe、yt-dlp 是否可用。

当前实现入口：

- `frontend/` 已初始化为 v2 专用 Vite + React + TypeScript 工程。
- 首屏保留 Legacy 的桌面式空间关系：左侧导航、中间窗口化下载工作台、右侧环境状态与迁移边界。
- `docs/UI_API_CONTRACT.md` 记录前端与未来本地 API 适配层的最小契约。

暂缓：

- 完整账号体系、云端同步、复杂多用户权限。
- 大型素材库管理。
- AI 分析工作台。
- 桌面打包，直到本地 Web UI 的工作流稳定。

## 5. 验收标准

- 用户能沿用 Legacy 的主要视觉记忆完成下载任务。
- 前端没有复制 `fetch`、`encode`、`subtitle` 等媒体业务逻辑。
- 下载任务失败时能显示可读错误，并保留失败清单。
- 本地验证包含前端构建/测试入口；如尚未引入前端工程，则本文档作为技术栈评估准入条件。
