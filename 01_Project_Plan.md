# MediaTools v2 Project Plan

## 1. 项目定位

MediaTools v2 是一个跨平台媒体工具集重构版，目标是在保留 Legacy 版本中被验证过的核心媒体工作流基础上，重新建立更小、更清晰、更可测试的实现。

### 核心目标
- **小巧精悍**：优先实现高频核心功能，避免一次性搬运旧架构。
- **跨平台优先**：Windows / macOS / Linux 使用统一路径、配置和命令约定。
- **依赖最小化**：优先标准库和系统工具，仅在必要时引入第三方依赖。
- **可测试可维护**：核心逻辑与 I/O、外部进程、UI 适配层分离。

### 非目标
- 不直接复制 Legacy 的 vendor 目录。
- 不在核心工作流和接口契约稳定前初始化完整前后端架构。
- 不提交运行日志、缓存、数据库、构建产物或本地工具配置。

## 3. 技术栈决策框架

技术栈已确认采用 Python CLI 优先；Phase 3 开始评估本地轻前端，但必须贴近 Legacy 的技术栈、布局结构和视觉习惯，以降低用户学习成本。单文件分发推迟到发布准备阶段评估。候选方案记录如下：

| 方案 | 优势 | 风险 | 适合场景 |
| --- | --- | --- | --- |
| Python CLI | 媒体处理生态成熟，脚本集成成本低，跨平台路径标准库完善 | 打包分发需额外设计，GUI/Web 需另行选择 | 以 CLI 和批处理为主 |
| Node.js / Vite 前端 | 若 Legacy 前端同源，恢复旧 UI 布局和组件习惯成本低；前端生态成熟 | 需避免重新引入 Legacy 的 vendor 与复杂服务边界 | 本地 Web UI、未来桌面壳或 Electron/Tauri |
| Go CLI | 单文件分发强，性能和并发好，跨平台构建稳定 | 媒体生态和快速脚本开发不如 Python | 强调独立二进制和部署 |

### 决策问题
1. 首批用户主要使用 CLI、桌面 GUI，还是 Web UI？**当前决策：CLI 已完成首批 MVP；Phase 3 评估 Legacy 风格本地轻前端。**
2. 是否需要单文件可执行程序分发？**当前决策：暂不作为 Phase 1 目标。**
3. 是否接受用户本机预装 ffmpeg？**当前决策：接受，作为外部系统工具探测。**
4. 首批核心功能是下载、转码、字幕、截图、批处理，还是 Adobe/外部工具联动？**当前决策：首批 MVP = probe / 转码 / 字幕转换 / 截图 / 下载；Phase 3 优先完善下载落地。**
5. 更看重快速迁移，还是长期分发和稳定部署？**当前决策：先快速建立可测试核心。**

## 4. 验收标准

### 治理初始化完成
- `AGENTS.md` 和 `01`-`05` 文档完整、互相引用有效。
- `03_Context.md` 能反映当前状态，不提前打钩未验证事项。
- `.gitignore` 能屏蔽运行产物、缓存、vendor、构建输出和本地配置。

### 技术栈初始化完成
- 仓库包含最小可运行入口。
- 仓库包含依赖声明、测试入口、格式化/检查命令。
- 首个测试能在干净环境中运行。
- README 写明安装、运行和验证命令。

### Git 工作流规范
- 所有 AI 辅助开发的提交需附带 `Co-Authored-By` 标记，用于追踪各工具贡献。
- 项目提供跨平台 Git hook（`.git-hooks/`）自动检测 AI 工具并注入 co-author。
- 提交前确保本地客观验证通过；推送后以 CI 绿灯作为跨平台验收依据。
- 详见 [`docs/AI_COAUTHOR_SETUP.md`](docs/AI_COAUTHOR_SETUP.md) 与 `AGENTS.md` 的 Git Commit 规范。

## 5. 风险清单

- **旧版残留污染**：工作区中可能存在 Legacy 缓存、vendor、数据库和构建产物，需避免误提交。
- **文档状态漂移**：治理文件若不及时同步，会误导后续 AI 或人工接手。
- **前后端过早复杂化**：前后端分离本身不是问题；核心契约未稳定前引入完整 Web / 桌面架构，会放大适配和维护成本。Phase 3 轻前端应先服务下载工作台，并保持 Legacy 布局与用户路径一致。
- **跨平台验证不足**：路径、外部命令、编码和权限问题必须从 Phase 1 开始测试。
- **代码规模失控**：单文件超过 500 行通常意味着职责混杂，需按 core / command / adapter / test fixture 边界拆分。
